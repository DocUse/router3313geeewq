# Bitrix Taxi Router

Серверное приложение для Bitrix24, которое распределяет лиды между рекрутерами по очереди и автоматически передает карточку следующему рекрутеру, если она не была взята в работу вовремя.

## Что уже реализовано

- хранение подключенных порталов и OAuth-токенов;
- регистрация и хранение групп распределения;
- список участников очереди по группе;
- обработка событий `ONCRMLEADADD` и `ONCRMLEADUPDATE`;
- первичное назначение лида на первого рекрутера;
- повторная передача лида следующему рекрутеру по таймеру;
- остановка цепочки, когда лид ушел из первоначального этапа;
- API для групп, сотрудников и ручного запуска таймерного воркера;
- UI для настройки групп через браузер;
- подхват контекста портала при открытии интерфейса внутри Bitrix24;
- локальные unit-тесты на ключевую бизнес-логику.

## Локальный запуск

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -e .
cp .env.example .env
python3 run.py init-db
python3 run.py serve
```

## Деплой на Fly.io

В проект добавлены:

- `Dockerfile`
- `fly.toml`
- `.dockerignore`

Что важно:

- имя приложения на Fly должно быть глобально уникальным;
- если `bitrix-router` не создается, используйте что-то вроде `docuse-bitrix-router-prod` или `docuse-bitrix-router-01`;
- приложение слушает `8080`;
- health-check идет в `/health`;
- SQLite вынесен в `/data/bitrix_taxi_router.sqlite3`, поэтому для нормальной работы нужен volume на Fly.

Перед деплоем проверьте и при необходимости поменяйте строку `app = "..."` в `fly.toml`.

## Основные endpoint'ы

- `GET /ui/groups` — экран настройки групп
- `POST /ui/groups` — тот же экран, но с автоподхватом контекста портала из Bitrix24
- `POST /install/callback` — сохранить портал и токены после установки приложения
- `POST /events/bitrix` — обработка событий лида
- `GET /api/groups?member_id=...` — список групп
- `POST /api/groups` — создать или обновить группу
- `DELETE /api/groups/{group_id}?member_id=...` — удалить группу
- `GET /api/employees?member_id=...` — список сотрудников портала
- `POST /api/jobs/reassign` — один проход таймерного перераспределения

## Важное допущение MVP

Лид считается взятым в работу, если его поле `STATUS_ID` больше не равно первоначальному этапу группы распределения.
