# ТЗ и план поэтапной разработки `Распределения сделок`

## Назначение документа

Этот документ нужен как рабочее ТЗ для разработки приложения по этапам.

Цель:
- не пытаться реализовать все сразу;
- двигаться небольшими безопасными шагами;
- на каждом этапе проверять, что уже сделанное действительно работает;
- снижать риск багов в логике распределения, автопередачи и интеграции с Bitrix24.

Документ опирается на:
- текущую спецификацию `distribution_deals_spec.md`;
- фактическую архитектуру проекта `bitrix_taxi_router`;
- документацию Bitrix24 по установке приложений, событиям CRM и REST API.

---

## 1. Цель MVP

Нужно сделать приложение Bitrix24 для раздела `Распределение сделок`, которое:
- устанавливается и открывается внутри Bitrix24;
- позволяет настроить одну группу распределения на портал;
- автоматически назначает новые сделки на выбранных сотрудников;
- передает сделку следующему сотруднику, если предыдущий не взял ее в работу за заданное время;
- учитывает лимиты нагрузки;
- если все участники заполнены по лимиту, не теряет сделку, а ставит ее в ожидание и перепроверяет доступность через отдельный интервал.

В первой версии не нужно:
- делать несколько групп распределения одновременно;
- строить сложную аналитику и историю в интерфейсе;
- добавлять лишние режимы распределения;
- пытаться закрыть все будущие сценарии заранее.

---

## 2. Как каждая функция будет реализована через Bitrix24

### 2.1. Установка приложения

Используются:
- install page приложения;
- install callback;
- `BX24.installFinish()`.

Что должно происходить:
1. Bitrix открывает установочный экран приложения.
2. Приложение получает контекст портала.
3. Backend сохраняет портал, домен и токены.
4. Установка завершается через `BX24.installFinish()`.
5. Пользователь попадает в основной интерфейс приложения.

### 2.2. Регистрация событий CRM

Используются:
- `event.bind`
- события:
  - `ONCRMDEALADD`
  - `ONCRMDEALUPDATE`

Назначение:
- `ONCRMDEALADD` нужен для первичного назначения новой сделки;
- `ONCRMDEALUPDATE` нужен для отслеживания изменений и остановки автопередачи, когда сделку взяли в работу.

### 2.3. Получение сделки

Использовать:
- `crm.item.get`

Параметр:
- `entityTypeId = 2` для сделок.

Назначение:
- получить актуальный статус сделки;
- получить текущего ответственного;
- читать нужные поля перед назначением и автопередачей.

### 2.4. Обновление сделки

Использовать:
- `crm.item.update`

Параметр:
- `entityTypeId = 2`.

Назначение:
- назначить нового ответственного;
- при необходимости записывать пользователя в выбранное поле ответственного.

### 2.5. Получение списка сделок для расчета нагрузки

Использовать:
- `crm.item.list`

Назначение:
- собрать сделки сотрудника;
- отфильтровать сделки по статусам нагрузки;
- посчитать текущую фактическую загрузку.

### 2.6. Получение стадий CRM

Использовать:
- `crm.status.list`

Назначение:
- заполнить поле `Статус для распределения`;
- заполнить блок `Статусы для определения нагрузки`.

### 2.7. Получение пользователей портала

Использовать:
- `user.get`

Назначение:
- загрузить список сотрудников;
- отобразить их в интерфейсе;
- позволить выбрать участников распределения.

## 3. Внутренняя архитектура, от которой идем

Разработка должна быть разделена на 5 частей.

### 3.1. Install/Auth layer

Отвечает за:
- установку приложения;
- сохранение портала;
- сохранение и обновление токенов;
- связь с конкретным Bitrix24-порталом.

### 3.2. Reference data layer

Отвечает за:
- пользователей;
- стадии CRM;
- доступные поля ответственного;
- синхронизацию справочников из Bitrix24.

### 3.3. Distribution config layer

Хранит:
- одну группу распределения;
- участников;
- лимиты;
- статус распределения;
- статусы нагрузки;
- поле ответственного;
- таймер автопередачи;
- интервал фоновой перепроверки;
- флаг активности.

### 3.4. Runtime engine

Отвечает за:
- прием событий из Bitrix24;
- решение о назначении;
- запуск и обработку таймеров;
- повторную передачу сделки;
- ожидание свободного участника, если все переполнены.

### 3.5. UI layer

Отвечает за:
- отображение настроек;
- понятные названия полей;
- сохранение конфигурации;
- безопасную и поэтапную работу пользователя с приложением.

---

## 4. Основные правила реализации

- Делать одну группу распределения на портал в первой версии.
- Делать один рабочий сценарий, а не несколько сразу.
- Каждая новая функция добавляется только после проверки предыдущей.
- Сначала добиваться стабильной серверной логики, потом усложнять UI.
- Не смешивать пользовательские таймеры и технические интервалы.
- Не делать скрытые важные параметры, если пользователь должен ими управлять.
- Историю автопередач пока хранить внутренне, без отдельного интерфейсного блока.

---

## 5. Поэтапный план разработки

## Этап 0. Стабильный каркас приложения

### Цель

Убедиться, что приложение стабильно ставится в Bitrix24 и открывается внутри портала.

### Что должно быть готово

- `/health`
- install flow
- install callback
- основной UI route
- сохранение портала и токенов
- корректное завершение установки

### Что проверяем

- установка проходит без черного экрана и без JSON вместо интерфейса;
- приложение открывается внутри Bitrix;
- после установки можно снова зайти в приложение.

### Критерий готовности

Приложение можно установить и открыть в Bitrix24 без ошибок.

---

## Этап 1. Чтение справочников из Bitrix24

### Цель

Научиться стабильно подтягивать данные портала без логики распределения.

### Что реализуем

- загрузка пользователей;
- загрузка стадий CRM;
- загрузка доступных полей ответственного;
- вывод этих данных в интерфейсе.

### Что проверяем

- UI показывает реальных сотрудников;
- UI показывает реальные стадии сделок;
- данные берутся из установленного портала, а не из моков.

### Критерий готовности

Пользователь открывает приложение и видит реальные данные портала.

---

## Этап 2. Сохранение одной группы распределения

### Цель

Сделать первую рабочую конфигурацию без запуска runtime-механики.

### Что реализуем

- название группы;
- тип распределения;
- участники;
- лимиты;
- событие;
- статус распределения;
- статусы нагрузки;
- поле ответственного;
- время ожидания;
- интервал перепроверки при переполнении лимитов;
- флаг активности.

### Что проверяем

- группа сохраняется;
- после перезагрузки все значения поднимаются обратно;
- UI отображает именно сохраненную конфигурацию;
- включение и выключение группы сохраняется.

### Критерий готовности

Одна группа на портал сохраняется стабильно и без потери данных.

---

## Этап 3. Первичное назначение новой сделки

### Цель

Сделать только первичное автоматическое назначение без автопередачи.

### Что реализуем

- прием `ONCRMDEALADD`;
- чтение сделки из Bitrix;
- проверку соответствия статусу распределения;
- проверку активности группы;
- проверку наличия участников;
- расчет нагрузки;
- выбор подходящего участника;
- запись ответственного в сделку.

### Что проверяем

- новая сделка в нужном статусе получает ответственного;
- если статус не подходит, приложение ничего не делает;
- если группа выключена, приложение ничего не делает;
- если лимиты достигнуты, система не падает.

### Критерий готовности

Первичное назначение работает корректно и стабильно.

---

## Этап 4. Автопередача по бизнес-таймеру

### Цель

Добавить механику передачи сделки следующему участнику, если ее не взяли в работу.

### Что реализуем

- хранение runtime-состояния по сделке;
- запуск таймера на значение `Время ожидания в секундах`;
- перепроверку текущего статуса сделки;
- остановку, если сделка ушла из статуса распределения;
- повторный подбор участника;
- повторное назначение.

### Что проверяем

- если сделка уходит в другой этап, автопередача прекращается;
- если сделка остается в статусе распределения, идет повторное назначение;
- цикл повторяется корректно.

### Критерий готовности

Сделка реально передается по цепочке участников, пока кто-то не возьмет ее в работу.

---

## Этап 5. Поведение при переполнении лимитов

### Цель

Реализовать отдельную механику ожидания, когда все участники достигли лимитов.

### Что реализуем

- состояние ожидания доступного участника;
- отдельный технический интервал перепроверки;
- повторные попытки назначения;
- понятную отдельную настройку этого интервала в интерфейсе.

### Что проверяем

- если все переполнены, сделка не исчезает;
- приложение не назначает сделку сверх лимита;
- как только кто-то освобождается, сделка назначается.

### Критерий готовности

Сделка не теряется даже тогда, когда все участники заняты.

---

## Этап 6. Служебная диагностика

### Цель

Сделать систему понятной для отладки и поддержки.

### Что реализуем

- внутренний журнал назначений;
- внутренний журнал автопередач;
- внутренние причины остановки;
- техническое состояние сделки в обработке.

### Что проверяем

- по логам можно понять, почему сделка назначилась;
- по логам можно понять, почему сделка не назначилась;
- по логам можно понять, почему автопередача остановилась.

### Критерий готовности

Поведение приложения объяснимо и отлаживаемо.

---

## Этап 7. Доработка UI

### Цель

Сделать интерфейс понятным и безопасным для пользователя.

### Что реализуем

- аккуратное разбиение формы на смысловые блоки;
- понятные подписи полей;
- подсказки под сложными настройками;
- отдельное объяснение разницы между:
  - таймером автопередачи;
  - интервалом перепроверки при переполнении лимитов.

### Что проверяем

- пользователь понимает, что настраивает;
- пользователь не путает бизнес-таймер и технический интервал;
- форма не выглядит перегруженной.

### Критерий готовности

Интерфейс можно использовать без технического объяснения со стороны разработчика.

---

## 6. Порядок, в котором нельзя прыгать

Правильная последовательность:

1. Установка и открытие в Bitrix24
2. Чтение справочников
3. Сохранение одной группы
4. Первичное назначение
5. Автопередача по таймеру
6. Перепроверка при переполнении лимитов
7. Диагностика
8. Полировка UI

Нельзя:
- пытаться делать красивый сложный интерфейс до стабильной бизнес-логики;
- делать автопередачу, пока не работает первичное назначение;
- делать сложные сценарии, пока не завершен путь одной группы на портал.

---

## 7. Что считать готовой первой версией

Первая версия считается готовой, если:
- приложение стабильно устанавливается в Bitrix24;
- одна группа распределения сохраняется;
- новая сделка в нужном статусе автоматически получает ответственного;
- если сделка не взята в работу вовремя, она уходит следующему участнику;
- если все переполнены, сделка ждет и перепроверяется;
- пользователь понимает настройки интерфейса.

---

## 8. Практический принцип работы над проектом

На каждом этапе действуем одинаково:

1. Сначала реализуем только один ограниченный кусок.
2. Потом вручную проверяем его на живом портале Bitrix24.
3. Только после этого двигаемся дальше.
4. Не добавляем новую механику, пока не подтверждена стабильность предыдущей.

Это основной способ избежать каскадных ошибок и трудноуловимых багов.
