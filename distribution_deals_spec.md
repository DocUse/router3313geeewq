# Спецификация раздела `Распределение сделок`

## Назначение раздела

Раздел `Распределение сделок` нужен для настройки автоматического назначения новых CRM-сделок на выбранных сотрудников Bitrix24.

Цель механики:
- определить группу сотрудников, между которыми распределяются сделки;
- задать правила выбора следующего ответственного;
- ограничить нагрузку на каждого участника;
- автоматически передавать сделку следующему участнику, если предыдущий не успел взять ее в работу за заданное время.

В рамках текущей версии раздела настраивается один тип логики:
- `По очереди с лимитами по нагрузке и времени`.

## Общий сценарий работы

1. Пользователь создает группу распределения и задает ее параметры.
2. Приложение получает из Bitrix24 пользователей установленного портала, стадии CRM и доступные поля ответственного.
3. Когда в Bitrix24 происходит событие `Создание новой сделки`, приложение проверяет, попала ли сделка в выбранный статус распределения.
4. Если попала, приложение выбирает подходящего участника по правилам нагрузки и очередности и назначает его ответственным.
5. После назначения запускается ожидание на заданное число секунд.
6. Если за это время сделка ушла из статуса распределения в любой другой этап, считается, что ее взяли в работу, и автопередача останавливается.
7. Если сделка осталась в статусе распределения, она автоматически передается следующему подходящему участнику.
8. Если свободных участников по лимиту нет, приложение не назначает сделку сразу, а продолжает повторно проверять доступность через отдельный короткий технический интервал.

## Термины

- `Группа распределения` — набор правил и участников, по которым работает автоподбор ответственного.
- `Статус распределения` — этап CRM, при попадании в который сделка должна быть взята приложением в обработку.
- `Статусы нагрузки` — этапы CRM, сделки в которых считаются активной нагрузкой для участника.
- `Взять в работу` — перевести сделку из статуса распределения в любой другой этап.
- `Лимит` — максимальное число активных сделок у участника в выбранных статусах нагрузки.
- `Технический интервал повторной попытки` — короткий внутренний интервал перепроверки доступности участников, если все достигли лимита.

## Описание полей формы

### 1. Наименование группы распределения

Назначение:
- человекочитаемое имя группы.

Что делает:
- позволяет пользователю отличать одну группу распределения от другой;
- используется в интерфейсе и в списке групп.

Правила:
- поле обязательно для заполнения;
- значение задается вручную пользователем;
- приложение не генерирует название автоматически, кроме возможной стартовой подсказки в UI.

### 2. Тип распределения

Назначение:
- определяет алгоритм назначения сделки.

Что делает:
- задает режим обработки новых сделок для данной группы.

Доступные значения:
- на текущем этапе только одно значение: `По очереди с лимитами по нагрузке и времени`.

Правила:
- поле обязательно;
- пользователь видит выпадающий список, но выбрать может только этот единственный режим;
- поле оставляется как dropdown на будущее расширение логики.

### 3. Участники распределения

Назначение:
- определяет сотрудников, между которыми приложение может распределять сделки.

Источник данных:
- пользователи Bitrix24 того портала, в который установлено приложение.

Что должно делать приложение:
- после установки приложения получить список пользователей портала;
- отобразить их в блоке участников;
- для каждого пользователя показать чекбокс выбора и индивидуальное поле лимита.

Правила:
- в распределении участвуют только сотрудники с отмеченными чекбоксами;
- неотмеченные сотрудники не рассматриваются алгоритмом;
- у каждого выбранного сотрудника должен быть лимит нагрузки;
- система должна хранить порядок участников как отдельную настройку очереди, даже если в интерфейсе он пока не вынесен явно.

### 4. Массовое заполнение лимита

Назначение:
- ускоряет установку одинакового лимита для нескольких участников.

Состав блока:
- переключатель области применения:
  - `Выбранных`;
  - `Всех`;
- поле `Лимит`;
- кнопка `Применить`.

Что делает:
- значение из поля лимита копируется в лимиты участников по выбранной области применения.

Правила:
- если выбрано `Выбранных`, значение применяется только к участникам с отмеченными чекбоксами;
- если выбрано `Всех`, значение применяется ко всем участникам блока;
- массовое применение не сохраняет группу само по себе, а только обновляет форму;
- фактическое сохранение происходит по нижней кнопке `Применить`.

### 5. Отслеживаемое событие

Назначение:
- определяет, на какое событие CRM реагирует группа распределения.

Доступные значения:
- на текущем этапе только `Создание новой сделки`.

Что делает:
- включает реакцию группы на создание новой CRM-сделки в Bitrix24.

Правила:
- поле остается выпадающим списком для будущего расширения;
- в текущей версии поддерживается только одно событие;
- при сохранении группа должна явно хранить, что работает по событию создания сделки.

### 6. Статус для распределения

Назначение:
- определяет этап CRM, при котором приложение должно запускать механику назначения.

Источник данных:
- стадии CRM-сделок текущего портала Bitrix24.

Что должно делать приложение:
- получить список этапов воронки сделок из CRM;
- показать их пользователю в виде выпадающего списка;
- сохранить выбранный этап как триггер распределения.

Правила:
- поле обязательно;
- если новая сделка создается сразу в этом статусе, приложение начинает алгоритм назначения;
- если сделка после назначения уходит из этого статуса в любой другой, автопередача считается завершенной, так как сделку взяли в работу.

### 7. Статусы для определения нагрузки

Назначение:
- определяет, какие стадии CRM считаются активной нагрузкой для участников.

Источник данных:
- стадии CRM-сделок текущего портала Bitrix24.

Что должно делать приложение:
- показать список стадий с чекбоксами;
- дать пользователю отметить те статусы, которые считаются активной загрузкой менеджера.

Правила:
- в нагрузку входят все сделки ответственного в отмеченных стадиях;
- учитываются все такие сделки сотрудника, независимо от того, назначило их текущее приложение или кто-то другой;
- поле влияет только на расчет доступности участника, а не на факт остановки автопередачи по конкретной сделке.

Пояснение для интерфейса:
- в блоке должно присутствовать описание:
  - `Вы выбрали режим распределения по нагрузке на менеджера, поэтому в данном поле следует указать, в каких стадиях активные сделки являются нагрузкой для выбранных менеджеров.`

### 8. Поле для установки/определения ответственного

Назначение:
- определяет, в какое поле сделки приложение записывает выбранного сотрудника.

Источник данных:
- поля CRM-сделки, позволяющие установить ответственного.

Поведение:
- по умолчанию используется стандартное поле `Ответственный`;
- при необходимости пользователь может выбрать другое поддерживаемое поле.

Правила:
- значение по умолчанию должно быть установлено сразу;
- если пользователь не меняет поле, применяется стандартный ответственный;
- приложение должно использовать выбранное поле как для первичного назначения, так и для последующих автопередач.

### 9. Время ожидания в секундах

Назначение:
- определяет, сколько времени текущий ответственный получает на реакцию до автопередачи следующему участнику.

Что делает:
- после назначения сделки запускается таймер ожидания;
- если сделка не ушла из статуса распределения за это время, приложение передает ее следующему подходящему участнику.

Правила:
- значение задается пользователем в секундах;
- это пользовательский бизнес-таймер;
- это не технический интервал фоновой перепроверки, когда все участники уже переполнены по лимитам.

### 10. Нижняя кнопка `Применить`

Назначение:
- сохраняет всю настроенную группу распределения.

Что делает:
- валидирует форму;
- сохраняет группу, участников, лимиты и остальные параметры;
- активирует группу для дальнейшей обработки событий.

Правила:
- без нажатия нижней кнопки изменения в форме не считаются сохраненными;
- после сохранения группа должна стать доступной для использования алгоритмом распределения.

## Алгоритм первичного назначения сделки

1. В Bitrix24 создается новая сделка.
2. Приложение получает событие `Создание новой сделки`.
3. Приложение определяет, есть ли активные группы, работающие по этому событию.
4. Для каждой подходящей группы проверяется:
   - совпадает ли текущий статус сделки со статусом распределения;
   - есть ли выбранные участники;
   - заданы ли лимиты и остальные обязательные параметры.
5. Для каждого участника считается текущая нагрузка:
   - берутся все его сделки;
   - отбираются только сделки в выбранных статусах нагрузки;
   - итоговое число сравнивается с лимитом участника.
6. Из участников, не достигших лимита, выбирается кандидат:
   - сначала по минимальной нагрузке;
   - при равной нагрузке — тот, кто дольше всех не получал новую сделку;
   - если и это значение одинаковое, как технический fallback используется стабильный порядок участников в группе.
7. Сделка назначается выбранному участнику в выбранное поле ответственного.
8. Для сделки запускается ожидание на значение из поля `Время ожидания в секундах`.

## Алгоритм автопередачи по таймеру

1. После первичного назначения запускается таймер ожидания.
2. По истечении времени приложение проверяет текущий статус сделки.
3. Если сделка уже ушла из статуса распределения, процесс завершается: сделка считается взятой в работу.
4. Если сделка все еще находится в статусе распределения, приложение снова подбирает подходящего участника.
5. Следующий участник определяется по тем же правилам нагрузки и выбора, что и при первом назначении.
6. Если найден доступный участник, сделка назначается ему, и таймер ожидания запускается заново.
7. Цикл повторяется до тех пор, пока сделка не выйдет из статуса распределения.

## Поведение при переполнении лимитов

Если все участники группы достигли лимита:
- сделка не назначается сотруднику сверх лимита;
- приложение не завершает процесс ошибкой;
- приложение переводит сделку в состояние ожидания доступного участника;
- далее приложение циклически повторяет проверку участников через отдельный короткий технический интервал.

Важно:
- этот интервал не равен полю `Время ожидания в секундах`;
- это отдельная настройка, которая должна быть показана в интерфейсе;
- в интерфейсе она должна быть названа понятным для пользователя образом, чтобы было ясно, что речь идет не о времени автопередачи сделки, а о частоте фоновой перепроверки доступности участников;
- это технический механизм фоновой перепроверки, но он должен быть управляемым из UI.

Пояснение для интерфейса:
- в форме должен быть отдельный параметр с понятным названием, например по смыслу `Интервал перепроверки, если все участники заполнены по лимиту`;
- рядом с параметром должно быть короткое пояснение, что эта настройка определяет, как часто приложение повторно проверяет, освободился ли кто-то из участников, когда на момент назначения все достигли лимита;
- эта настройка не должна смешиваться с полем `Время ожидания в секундах`, которое отвечает за автопередачу уже назначенной сделки.

## Правила расчета нагрузки

Нагрузка участника считается так:
- берутся все сделки, где этот сотрудник указан в выбранном поле ответственного;
- из них учитываются только сделки в выбранных статусах нагрузки;
- итоговое число сравнивается с индивидуальным лимитом этого участника.

Следствия:
- приложение считает не только сделки, которые ранее назначило само;
- нагрузка отражает реальную занятость сотрудника в CRM в рамках выбранных стадий;
- один и тот же сотрудник может быть исключен из назначения в одной группе из-за своей общей CRM-нагрузки.

## Правила выбора следующего менеджера

Основной принцип выбора:
- в первую очередь выбирается сотрудник с наименьшей текущей нагрузкой.

Если нагрузка одинакова:
- приоритет получает сотрудник, который дольше всех не получал новую сделку через приложение.

Если после этого равенство все еще сохраняется:
- используется детерминированный технический fallback по порядку участников в группе.

## Что должно подтягиваться автоматически из Bitrix24

Приложение должно уметь автоматически получать из установленного портала:
- список пользователей для блока `Участники распределения`;
- список стадий CRM-сделок для поля `Статус для распределения`;
- список стадий CRM-сделок для блока `Статусы для определения нагрузки`;
- список доступных полей сделки, которые могут использоваться для установки ответственного.

## Зафиксированные решения перед кодом

1. Короткий технический интервал повторной попытки назначения не должен быть жестко зашит в код.
   - Он должен быть вынесен в интерфейс как отдельная настраиваемая настройка.
   - Название настройки должно быть понятным пользователю и явно отличаться от бизнес-таймера автопередачи сделки.

2. Группа распределения должна иметь флаг активности `вкл/выкл`.
   - Эта настройка должна существовать в модели и быть доступна для управления.

3. В первой версии достаточно одной группы распределения на портал.
   - Поддержку нескольких групп можно оставить на будущее расширение.
   - Текущее подтвержденное поведение реализации: при сохранении второй группы на том же портале первая не добавляется второй карточкой, а замещается новой конфигурацией.
   - Причина не в UI-отрисовке, а в текущем ограничении модели и runtime: на портал хранится одна конфигурация группы, и обработка события `ONCRMDEALADD` читает одну группу.
   - При возврате к этой задаче нужно отдельно спроектировать мультигрупповой режим:
     - как хранить несколько групп на один портал;
     - как отображать и редактировать их в интерфейсе;
     - как выбирать группу при обработке новой сделки, если активных групп несколько;
     - что считать допустимой и недопустимой конфигурацией при пересечении условий групп.

4. Историю автопередач пока можно оставить фоновой внутренней логикой.
   - На текущем этапе не требуется выводить ее как отдельный пользовательский блок интерфейса.

## Результат работы раздела

После настройки и сохранения группы раздел должен обеспечивать следующее поведение:
- новая сделка в нужном статусе автоматически получает ответственного;
- ответственный выбирается с учетом лимита и текущей нагрузки;
- если сделка не взята в работу вовремя, она уходит следующему подходящему участнику;
- цикл продолжается, пока сделка не выйдет из статуса распределения;
- если доступных участников нет, приложение продолжает ждать и технически перепроверять доступность без потери сделки.
